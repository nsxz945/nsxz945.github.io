Chapter 2: this All Makes Sense Now!

this丢失

callback会造成成this的丢失。

显式绑定this

obj.foo()这样的调用可以隐式绑定this。

call(..) 和 apply(..)可以用来绑定this。当参数是简单数据类型string, boolean, number时，是将它们的对象形式(相当于由new String(..), new Boolean(..), or new Number(..)生成的对象)作为this。

Hard Binding

可以用来解决this丢失的问题。因为this被明确的确定，所以称为hard binding。

####实现hard binding的三种常用方法

###new Binding

使用new操作符时，经过如下操作：

  1. 创建一个新的对象
  2. 这个新对象与构造函数通过原型链联系在一起(对象的proto指向构造函数的prototype)
  3. 新对象将作为构造函数执行时的this
  4. 除非构造函数返回一个指定对象，否则将默认返回这个新对象

hard binding的三种方法中，.bind方法生成的函数的this可以被new操作符复写，而另外两种则不会被复写。

摘自MDN关于bind的polyfill：

与文章中的版本相比，改进了Fbound函数中关于this的逻辑判断。

this的判断规则

逐条检查直至有一条符合。

  1. 函数是否由new进行调用。如果是，this指向新建的对象。
    var bar = new foo()
  2. 函数是否通过call，apply进行调用(显式绑定)。注意，显示绑定可能在hard binding中。如果是，this指向指明的对象。
    var bar = foo.call( obj2 )
  3. 是否是隐式绑定？this指向直接调用的对象。
    var bar = obj1.foo()
    4.其他情况，this默认指向global对象。在严格模式下，this为undefined。

例外

被略过的this

当传null或者undefined作为绑定参数给call，apply时，默认this会生效。

这个方法，可以用来传一个参数数列，或者进行curry化:

safer this

比起传null(this指向global)，传一个空对象作为this更为安全。
Object.create( null )，与{}创建空对象相比，空得更彻底，它不继承Object

Softening Binding

使用.bind()进行hard binding之后，无法再用bind改变它的this指向。而Softening Binding，使得我们可以实现这个需求。

Lexical this

ES6中引入箭头函数，箭头函数中的this无法被绑定。this在其中的表现，更像是作为一个普通变量，在作用域链链中逐级去查找最近的this。

‘[]’是key访问，’.’是property访问。在实际中，二者访问同一个地址，输出一样的值。二者最大的不同在于，’.’操作符要求输入符合规范的属性名，而’[]’能够接受任何UTF-8/unicode兼容的字符，以及变量。

Chapter 5: Prototypes

myObject.foo在原型链上的读写

  • 如果在高层原型链上，有foo属性，且不为只读，myObject.foo='bar'会直接在myObject上创建属性foo并赋值，称为shadowed属性

  • 如果在高层原型链上，有foo属性，且为read-only (writable:false),那么已存在属性的修改，以及在myObject上创建shadowed属性都是不允许的。

  • 如果在高层原型链上，有foo属性，且它有配置setter，那么setter总是会被调用。

insertMany 多数据添加

db.nameOfCollection.insertMany([JSONString1, JSONString2])
将对个document放入数组中作为参数，便可以进行多项插入。

另外，insertMany接受第二个参数作为设置项。insertMany([JSONString1, JSONString2],{"ordered":false}),使用inordered的方式插入多项数据，这样，当某几项因重复而报错时，它们仅会被跳过，而不会在错误的地方中止。

find 查询

db.nameOfCollection.find({"key": val}).pretty()
这样可以查询指定collection中key为val的document。当key的数据类型为数组时，则是查询数组中是否含有val。如果要对val在数据中的位置进行限定，键名需要指定位置，如db.nameOfCollection.find("key.0": "val").pretty()则是匹配数组中的第一项。

find() vs findOne()

find()用来查找document对应的指针，findOne()查找并返回document的内容

在设置时，键名表示要设置的字段，值表示是否显示(0为不显示，1为显示)。_id默认显示。

cursor.sort({field:1})
cursor.sort([[field,1],[field,-1]])
根据指定字段，对输出结果进行排序。
参数中，字段的值为1时，进行降序排列；-1时，进行升序排列。

cursor.skip(number) 跳过一定数量的docs

在实际使用中，sort,skip,limit的调用不会对最终结果造成影响。发起请求后，mongodb始终按照sort、skip、limit的顺序执行。
当sort文件数目过多时，mongodb会因为内存过大而报错。这种情况，我们要预先在shell中执行db.collection.createIndex({field:1})，对字段建立索引。

week5 Indexes and Performance

Index

创建索引，就是一笔交易。
付出：增加磁盘占用空间，增加写入时间。
收货：加快读取时间。

Multikey Indexes

创建的索引，是一种线性排列，是对数据的一次整理。
索引1（a,b,c）与索引2（b,a,c）是完全不同的。

  1. 索引1在只查询a时也可以生效，而索引2则不行。
  2. 查询(a,b)时，索引1能生效，而查询(a,c时)索引并不能生效。

这都是因为我们的索引中，次级键名仅在主键名的每个局部被按序整理

unique index

_id就是一个默认的唯一索引。

Sparse Index

在创建索引时，如果有的document中相应的键值对为空时，需要使用稀疏索引选项才能正常创建。

foreground & background

后台创建索引，将不会影响数据库本身的使用。但速度较慢。
前台创建索引，会阻止该数据库下其他即将到来的读写操作。但速度较快。

使用多个数据库协调工作，可以充分利用background Creation的特性。当主数据库需要创建索引时，使用备用数据库提供服务，当索引创建完成后，再使用主数据库提供服务，并对两个数据库进行同步。

在分析结果中，我们可以看到最终的执行方案winnerPlan，以及其他被淘汰的方案。

week7 Application Engineering

Write concern

为了更快的速度，我们读写数据是在内存中进行，并由日志(journal)将数据同步到数据库中。

通常的设置中，我们只对journal进行设置:1.false，请求无需等待journal写入磁盘。2.true，请求需要等待journal写入磁盘。

第一种速度更快；第二种数据更安全，宕机时，数据库可以从磁盘中的journal将数据复写出来。

Network error

网络请求失败，会导致数据库被写入，但app以为数据库没写入。这样APP就可能会发起多条重复的请求。（网络通常都很好，实际情况中该问题比较罕见）

比如update中的$inc就会带来错误的结果。解决办法是，用insert替代update操作。

Introduce to replication

最少需要3个。一个主数据库，其他为次要数据库。当主要数据库宕机后，会选择一个数据库作为主数据库。

数据交互是在主数据库中进行的。

write consistency

如果只允许APP读取主数据库，那么就可以保证读取的数据是最新的。

Replica Set Elections

集群结点类型

regular、arbiter、delayed regular、hidden

都可以参与promary node的投票，但后两种都不能成为primary node。
arbiter是为了保证一定能选出primary node。

rollback

当主数据库宕机重启后，主数据会作为次要数据库，同时把当前主数据库中未存数据以文件存储(留待手工处理），然后与当前主数据库保存一致。

Connecting to a Replica Set from the Node.js Driver

配置方法与单一数据库相同，只需要将数据库地址用逗号相连即可。

其实，只要Node.js Driver连接上集群中的一个数据库，它就能自动检测到其他数据库。但为了避免单一数据库无法连接的情况，我们还是应该把集群中数据库的地址都写上。

failover in the Node.js Driver

当主数据库宕机时，期间的数据库操作会被node.js Driver缓存。当Driver找到可用的数据库后，这些请求会依次被重新发起。
所以，我们不必担心这个问题。

#箭头函数的this指向

普通函数的this

以往对函数this的讨论，主要是针对由function(){}创建的函数。它的this很明确，哪一个对象调用了该函数，this便指向哪一个对象。默认由window进行调用。通过call，apply，bind可以改变函数的this。

箭头函数的this

简单的总结下箭头函数中this的指向：

  1. 箭头函数不绑定this,箭头函数中的this相当于普通变量。
  2. 箭头函数的this寻值行为与普通变量相同，在作用域中逐级寻找。
  3. 箭头函数的this无法通过bind，call，apply来直接修改。
  4. 改变作用域中this的指向可以改变箭头函数的this
  5. eg. function closure(){()=>{//code }}，在此例中，我们通过改变封包环境closure.bind(another)()，来改变箭头函数this的指向。
